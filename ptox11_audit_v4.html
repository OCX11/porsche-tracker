P1 — Data Loss or Broken Core Functionality

P1 AutoTrader crashes every scrape cycle — NameError: _PROXY_URL rennmarkt/scrapers/autotrader.py · rennmarkt/main.py

Every rennmarkt scrape cycle logs: ERROR ✗ AutoTrader: name '_PROXY_URL' is not defined. The split repo moved AutoTrader into rennmarkt/scrapers/autotrader.py but this file still references _PROXY_URL from the old proxy init pattern (DataImpulse). Since DataImpulse was retired and the proxy config no longer sets _PROXY_URL, the variable is never defined when AutoTrader is loaded. The scraper crashes at line 778 before returning any listings.

Result: 0 AutoTrader listings scraped. 958 active AutoTrader DB rows are now stale with no new listings coming in and no sold-marking firing (the 0-scrape guard prevents it). AutoTrader's 958 listings are ghost inventory.

FIX — rennmarkt/scrapers/autotrader.py
# Line 778 — old DataImpulse guard still present if not _PROXY_URL or not _PROXY_CFG.get("enabled"): log.info("AutoTrader: proxy disabled, skipping") return []
# AutoTrader now uses Scrapling (no proxy needed) # Remove the _PROXY_URL guard entirely, or replace with: # (no proxy check needed — Scrapling bypasses Akamai natively)
P1 2,963 active listings marked stale (not seen in 3+ days) — partial scrape guards blocking sold-marking rennmarkt/main.py · core/db.py mark_sold()

DB shows 551 AutoTrader + 868 DuPont + 1,544 eBay listings with date_last_seen older than 3 days. The partial scrape guard (fires when scraped count < 50% of active) is preventing mark_sold() from firing because each scrape cycle only returns a fraction of active inventory. Log confirms: "Only 18 cars scraped vs 916 active — skipping sold-marking" for DuPont and similar for eBay (49 vs 1,825).

This is a structural problem: scrape cycles are returning far fewer listings than the active DB count because we're scraping page 1 only in fast mode. The guard was designed to prevent accidental mass-sell, but it's now permanently tripped because DB counts have grown (4,122 total) while scrape depth hasn't increased proportionally. The result is zombie listings that will never be marked sold.

Why DuPont is so bad: 916 active vs 18 scraped = 2% of inventory per cycle. DuPont returns ~22 listings per API page and we may only be fetching 1 page. With 916 active records from a full bootstrap, fast-mode scraping can never clear stale records.

FIX — Two-pronged approach
# 1. Lower the sold-marking threshold for retail sources from 50% to a flat minimum # Instead of: if scraped < 0.5 * active: skip # Use: if scraped < MIN_SCRAPE_FLOOR[dealer]: skip # MIN_SCRAPE_FLOOR = {"DuPont Registry": 15, "eBay Motors": 40, "AutoTrader": 30, ...} # 2. Add a staleness-based cleanup job: any listing not seen in 14 days → mark sold # Run once daily in rennmarkt/main.py: # UPDATE listings SET status='sold' WHERE status='active' # AND date_last_seen < date('now', '-14 days') # AND dealer NOT IN ('Bring a Trailer','Cars and Bids','pcarmarket')
P1 RennMarkt index.html is a "Launching Soon" splash — real dashboard at dashboard.html docs/index.html · docs/dashboard.html

The public-facing docs/index.html (8 KB) is a splash/waitlist page that redirects logged-in users to /dashboard.html. The actual RennMarkt dashboard lives at docs/dashboard.html. This is intentional architecture (gated public launch) but means docs/dashboard.html is the file to audit for performance. Its size was not checked in this audit — if dashboard.html still embeds pre-built HTML card fragments (the old P1-2 architecture), it may still be large.

The v3 P1-2 fix (migrate to JSON + client-side render) appears to have resolved index.html size, but the question is whether dashboard.html inherited the fix or still uses the old pattern. Recommend running wc -c docs/dashboard.html immediately.

P2 — Wrong Data Shown or Major UX Failures

P2 auctions.html is 1.6 MB — nearly as bad as old index.html was rennauktion/build_dashboard.py · docs/auctions.html

docs/auctions.html is 1,644 KB for 57 auctions (~29 KB/card). This is on the same trajectory as the old index.html problem at 11.9 MB. Auction cards include embedded scatter graphs (comp dot plots), which are likely the bulk of the size — each card probably inlines all comp data points as inline JS/SVG rather than loading from external JSON.

With only 57 auctions, 1.6 MB is unsustainable. If auctions grow to 200+ (as planned with DuPont/Hagerty expansion), it'll be 5+ MB. The same JSON-data + client-side render approach used for the RennMarkt dashboard fix should be applied to build_dashboard.py.

P2 eBay VIN coverage at 17% — worse than v3's 23% despite being the largest source rennmarkt/scrapers/ebay.py · core/db.py

eBay Motors has 1,825 active listings but 1,507 (83%) have no VIN. This has worsened since v3 (was ~77% no-VIN). eBay's Browse API doesn't reliably return VIN for all listings — it's scraped from item detail pages. With eBay now at nearly 2x its previous count (924 → 1,825), the enrichment job may not be keeping up. This matters because: (1) VIN dedup is the most reliable match key; (2) VIN-less listings can't use the vin_history table; (3) generation decode fails without a VIN.

P2 Comp freshness at 24% within 6 months — FMV anchored to 2024 prices sold_comps table · rennauktion/comp_scraper.py

6,656 total comps. Only 1,617 sold within 6 months (24%). The v3 finding flagged this at 21% and estimated that daily comp scraping should produce ~4,500 fresh comps in the 6-month window. We've gone from 1,284 → 1,617 (+333 fresh in 12 days) which tracks with C&B backfill (487 total added) but most of those are older historical comps, not fresh ones. BaT has 6,049 of 6,656 total comps. The daily comp scraper needs to be running reliably. Check data/last_comp_scrape.txt — if it's missing runs, FMV accuracy is degrading.

The C&B backfill confirmed the scraper works. The question is why BaT's fresh-6-month count is only ~1,284 when BaT's daily auction volume for Porsches is 10–20 cars. At 15 cars/day × 180 days = 2,700 expected. Something is limiting the BaT scrape — either the daily scrape isn't running, or deduplication is silently skipping records.

P2 seen_alerts_push.json unbounded growth — 2,850 entries and climbing notify_push.py · rennauktion/notify_push.py

The push seen-alerts file has grown from 443 entries (April 19) to 2,850 entries (May 4) — a 6.4x increase in 15 days. At this rate it'll be 50K entries by year end. The 30-day prune runs on load but 2,850 entries suggests alerts are accumulating faster than they prune, or the prune isn't working correctly. At 397 KB now, it'll become a performance concern when it hits several MB and is loaded/parsed on every alert cycle.

Also: with 2 active push subscriptions instead of 1, the seen-alert count may be doubling because each alert fires once per subscription. Verify the dedup key doesn't depend on the subscriber endpoint.

P2 C&B scraper accepting Cayenne / Panamera / Macan — wrong model data reaching DB rennauktion/scrapers/cnb.py · shared/scraper_utils.py

The rennauktion log shows C&B scraping 22 total Porsche listings, then filtering to 12 (10 filtered out). The filtered logs include: Cayenne E-Hybrid Coupe, Cayenne Turbo Coupe, Panamera 4 Edition, Macan S, Macan Turbo. This is correctly filtering at the _is_valid() level, but these excluded cars may be reaching the DB temporarily before being filtered, or they're being rejected but still consuming scrape quota. The real concern: if the filter is in the rennauktion scraper but shared/scraper_utils.py _is_valid_listing() is slightly different, models could slip through in one path and not another. Verify both filter functions have identical exclusion lists.

P2 3 duplicate active listing URL groups in DB core/db.py upsert_listing()

The DB now has 3 URL groups with 2+ active listings sharing the same listing_url. In v3 this was 0. This may be from the repo split introducing a second code path that upserts with different dedup logic, or from BfB/Rennlist scrapers that don't normalize URLs consistently. Identify the specific URLs and their source to determine if this is a code bug or legitimate (e.g., a listing relisted with same URL).

P2 Cayman VIN decode failures — WP0AA2 series not decoding generation core/vin_decoder.py

Scrape log shows multiple failures: "2014 Cayman WP0AA2A86EK VIN=AA2 → FAILED (no decode)", "2011 Cayman WP0AA2A89BU VIN=AA2 → FAILED". The series code AA2 is being extracted but not mapping to a generation. This is the Cayman 987 / 981 series. The VIN decoder's Cayman lookup is missing the AA2 series code — it likely only handles CA2/CB2/CC2 from the reference table in HANDOVER.md. This means all base Cayman listings from eBay/AutoTrader get no generation, impacting FMV comp matching and filter chips.

Section Audits

A. FMV Engine

MAJOR UPGRADE FMV is now persisted to DB with 7 columns (fmv_value, fmv_confidence, fmv_comp_count, fmv_low, fmv_high, fmv_pct, fmv_updated_at). score_and_persist() runs every cycle on both rennmarkt (3,845 listings) and rennauktion (3,979 listings). This unlocks search-page FMV, deal-score alerts, and FMV history.

CONCERN Both rennmarkt AND rennauktion run score_and_persist() on the same shared DB every cycle. They run on different cadences (rennmarkt ~2 min, rennauktion ~5 min). This creates a write race condition — both processes update fmv_updated_at concurrently on the same listings table. SQLite's WAL mode should handle this without data corruption, but it's worth verifying WAL is enabled and monitoring for SQLITE_BUSY errors in the logs.

NOTE Comp freshness at 24% within 6 months means the FMV weighted median is still being pulled toward 1–2 year old prices. The HIGH confidence improvement (78% → 90%) is real and meaningful, but the underlying price accuracy issue remains until the comp scraper runs more reliably.

F. Data Quality

eBay VIN coverage at 17% is the biggest data quality issue. Without VINs, eBay listings can't use vin_history cross-source detection and generation decode is unreliable (falls back to year-only). The eBay scraper has an item-detail enrichment path — it's just not running fast enough vs volume.

Generation coverage at 95% is excellent for VIN-bearing sources. The remaining 5% are primarily PCA Mart, Rennlist, BfB, and eBay listings where no VIN is available.

Cayman VIN decode failures (WP0AA2 series) affect a meaningful subset — base Cayman 987/981 is a common sourced model. The VIN reference table needs AA2 → Cayman added.

G. Comp Scraper

H. Vestigial Code Risk

The project root still has the old main.py which imports the old root-level scrapers. If anything accidentally runs python3 main.py from the project root instead of python3 rennmarkt/main.py, it'll use the old code paths — potentially overwriting DB records with data from the non-split scrapers. The root main.py should be renamed or have a guard added at the top to refuse execution.

New Findings Not in v3

OBSERVATION eBay grew from 924 → 1,825 active listings — largest source by far, now 44% of all inventory eBay Motors

eBay is nearly double its v3 count and represents 44% of the 4,122 total active listings. This is structurally important: eBay has the worst VIN coverage (17%), the least reliable listing dedup (URL-based only without VIN), and produces the most noisy FMV signals (no trim on many listings, broad gen pool matching). As eBay grows, its data quality issues become more prominent in the overall FMV accuracy picture. Consider whether eBay volume should be capped or filtered more aggressively — e.g., excluding no-VIN, no-trim eBay listings from FMV scoring entirely.

OBSERVATION AutoTrader count ballooned: 8 → 958 active — but now completely broken AutoTrader

AutoTrader went from 8 active (v3, limited by Akamai blocks) to 958 after the Decodo/Scrapling migration. But then the proxy refactor broke the scraper with the _PROXY_URL NameError. AutoTrader has excellent data quality (100% mileage, 100% VIN, near-0% missing images) — it's the cleanest retail source. Fixing the scraper bug is high priority.

SUGGESTION vin_history cross_source events (310) are gold — surface these in the UI vin_history table

310 "cross_source" events means 310 VINs have appeared on multiple platforms. This is exactly the intelligence signal the system was built for — a car listed on DuPont AND eBay is almost certainly a dealer or distressed seller. These should be surfaced as a "Seen on multiple platforms" badge on listing cards and/or a dedicated dashboard section. The data exists; it just needs a UI surface.

SUGGESTION hagerty_valuations table exists in SQLite — not mentioned in any docs data/inventory.db

The DB has a hagerty_valuations table (from the old main.py --hagerty flag). It's not referenced in the current rennmarkt or rennauktion code. Either: (a) the old Hagerty scraper was archived during the split and the table is orphaned, or (b) a new Hagerty valuation feed needs to be wired into core/fmv.py as a fallback. Clarify status — if orphaned, remove from core/db.py init_db().
